# Plan: Name plan approval PDFs after their proposals

## Context and outcome

SASE materializes every tale and epic proposal inside its neutral approval bundle as `plan.md`. The Telegram outbound
integration renders that owned resource beside itself, so Telegram currently receives a document whose visible name is
always `plan.pdf`, even when the durable proposal has a descriptive name such as `sase_plan_improve_gate_controls.md`.
Plan-gate notifications already carry the durable source path in their trusted `original_plan_file` action metadata.

Use that metadata to name the Telegram document `sase_plan_improve_gate_controls.pdf` in this example. Apply the same
rule to both `PlanApproval` and `EpicApproval`, since both use the neutral `plan.md` resource. This work is independent
of the separate approval-control label change and has no dependency on it.

## Telegram attachment naming contract

- Keep SASE's neutral gate bundle and shared Markdown renderer unchanged: continue reading and converting the owned
  `plan.md` resource, and continue creating and deleting the same internal `plan.pdf` working file.
- At the Telegram transport boundary, derive an explicit PDF document filename from the basename of the non-empty
  `original_plan_file` path, replace only its final suffix with `.pdf`, and pass that name through the existing
  `send_document` filename support. Directory components must never become part of the transmitted name.
- Restrict the override to the rendered primary plan attachment of `PlanApproval` and `EpicApproval` notifications.
  Existing behavior for workflow-completion Markdown, launch previews, chat-response PDFs, already-PDF attachments,
  diffs, images, and other documents must remain unchanged.
- Treat naming metadata as presentation-only and best-effort. If the original plan path is absent, blank, malformed, or
  has no usable basename, preserve today's filename selection instead of failing the attachment or the approval message.
  Likewise, preserve the current Markdown-document fallback when PDF rendering is unavailable.
- Do not copy, rename, or render from the durable proposal path. The bundle resource remains the reviewed source of
  truth, so edits made through the approval gate and the existing temporary-file cleanup semantics are retained.

## Regression coverage and verification

- Add focused outbound coverage using a real command-backed plan gate whose durable proposal has a distinctive,
  multi-part filename. Assert that Telegram still converts the bundle's `plan.md` and sends the generated bytes with the
  proposal-derived `.pdf` filename rather than `plan.pdf`.
- Cover both tale and epic approval actions, and verify path-like original metadata is reduced to a safe basename.
- Add compatibility cases for missing or unusable `original_plan_file` metadata, confirming delivery falls back to the
  generated path's existing display-name behavior without raising.
- Retain or extend non-plan attachment assertions to prove the override does not rename ordinary converted Markdown or
  pre-existing PDF documents, and keep conversion-failure behavior covered.
- Run the focused outbound and command-backed gate tests while iterating. Before handoff, install the linked
  `sase-telegram` repository's development dependencies if needed and run its full `just check` suite.

## Ownership and risks

This is a Telegram presentation/transport change in the linked `sase-telegram` repository; SASE core already supplies
the durable plan identity and should not gain transport-specific naming logic. The main risks are accidentally using the
durable file as attachment content, leaking directory components into Telegram's filename, or applying one plan's name
to unrelated attachments. Constrain the resolver to the primary rendered plan resource, keep content sourced from the
verified bundle, and exercise the fallback and non-plan cases explicitly.
