# Plan: Restore Telegram outbound delivery for epic approval gates

## Context and diagnosis

The axe digest at `~/.sase/axe/error_digests/digest_20260717_082908.txt` contains 21 identical `telegram/tg_outbound`
subprocess exits between 08:18 and 08:28. The preceding 08:18 digest contains another 54 instances beginning at 07:49.
The full lumberjack output resolves the digest's intentionally terse `exit code 1` to this path:

1. `tg_outbound` loads the first unsent notification, an `EpicApproval` created at 07:48, just after Telegram's 07:47
   high-water mark.
2. The command-backed epic request correctly advertises only `epic`, `reject`, and `feedback` terminal choices.
3. `formatting._format_plan_approval()` loads the verified gate, but unconditionally calls
   `load_progress(..., default_choice_id="approve")` for both tale and epic requests.
4. A missing `telegram_gate_progress.json` is normal for a gate that has never been rendered. Recovery calls
   `initial_progress()` with `approve`; `gate_flow.select_choice()` rejects it because the epic envelope has no such
   choice, raising `ValueError: unknown gate choice: approve` before Telegram sends anything.
5. Formatting occurs before the outbound loop's send-error boundary and before `mark_sent()`. That preserves
   at-least-once delivery, but it also means the same poison notification is retried on every lumberjack tick. Axe
   notifications generated by those retries accumulate behind it, explaining the increasing batch size and repeated
   digests. The epic request was canceled at 07:55, after failures had already begun; cancellation is not the cause, and
   dismissed notifications intentionally remain eligible for Telegram delivery.

The regression is at the integration seam between command-backed tier-specific plan choices and the later shared
Telegram gate-progress feature. Existing epic-formatting coverage constructs only a lightweight `request.json` and does
not supply a verified `bundle_path`, so it exercises the legacy choice renderer rather than the crashing gate-progress
branch.

## Implementation

Work in the linked `sase-telegram` repository, opened through `sase repo`, and keep the host SASE plan-gate contract
unchanged: epic gates are correct to expose `epic` rather than `approve`.

- Make plan-gate progress initialization tier/choice aware. Tale `PlanApproval` gates should still initialize the
  advertised `approve` choice so its default `commit_plan` and `run_coder` add-ons render selected. Epic `EpicApproval`
  gates should not inject a tale-only choice; they need no preselected progress because they have no approval add-ons.
- Harden `gate_flow.load_progress()` so an absent, malformed, or stale progress file cannot turn a caller-provided
  default that is not present in the verified envelope into an exception. Normalize such a default to no selection,
  while retaining valid saved choices, filtering stale add-on IDs, and preserving message/chat metadata.
- Keep envelope-advertised buttons and callback behavior unchanged: epic renders `epic`, `reject`, and `feedback`; tale
  retains its compatibility presets and add-on submit flow. Do not hide the problem by advancing the high-water mark on
  an unsent notification or by broadly swallowing formatter failures.

## Regression coverage

- Add an end-to-end formatting regression built with SASE's real `create_plan_approval_gate()` for an epic plan and a
  verified bundle. Assert formatting succeeds from a missing progress file and renders only the epic envelope's three
  remote choices. Cover the canceled/dismissed-equivalent bundle state if that can be represented without coupling the
  test to user data.
- Add focused gate-progress tests for an unknown default on missing/malformed state and for stale persisted choice IDs;
  both must recover without raising or selecting a choice not advertised by the envelope.
- Preserve the existing tale-gate add-on test and strengthen it as needed to prove the two default-selected add-ons
  remain checked and submit correctly.
- Exercise the outbound path with the realistic epic notification under isolated notification/high-water fixtures,
  confirming it returns success and reaches `mark_sent()` without a real Telegram call. Never point a test or dry run at
  the user's live notification store, credentials, or high-water file.

## Validation

From the linked `sase-telegram` checkout, run `just install`, the focused tests covering gate progress, formatting, and
outbound integration, then `just check` (ruff, mypy, and the full pytest suite). Re-run the realistic epic regression
after the full suite to verify no order-dependent state leaked between tests. The live failure signature is resolved
when the exact `EpicApproval` choice set formats without `unknown gate choice: approve`; operational rollout of the
installed chop can then proceed through the normal repository/update workflow rather than mutating live SASE state
during testing.

## Risks and boundaries

- Default normalization must not erase valid saved selections or tale add-on defaults.
- Selecting `epic` implicitly would be misleading because rendering is not approval; leave epic progress unselected
  until the user taps a terminal choice.
- The outbound high-water design makes formatter failures deliberately visible and retryable. This plan fixes the
  invalid input supplied to progress recovery while preserving that delivery guarantee.
- No edits are needed in the primary `sase` repository, its memory files, or generated agent instruction files.
